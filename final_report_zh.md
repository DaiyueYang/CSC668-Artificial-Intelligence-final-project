# 基于 AI 的受约束旅行行程规划

## 1. 引言

本项目实现了一个受约束旅行行程规划原型。系统接收合成的景点数据和旅行参数，输出按天排列的访问计划。行程必须满足预算、每日可用时间、景点开放时间、参观时长、景点间交通时间、最大等待时间等硬约束，同时尽量提高总满意度。

本项目使用合成数据，不调用真实旅游平台、地图 API 或外部优化服务。核心规划逻辑由项目自行实现，主要包括一个贪心基线算法和一个 beam search 规划算法。`numpy` 和 `pandas` 用于数据生成和表格处理，`matplotlib` 用于结果可视化。

除了生成单个行程外，这个原型还包含一条可重复的实验流水线。命令行接口可以生成数据集、运行单一场景或全部场景、验证生成行程、输出可读的 Markdown 结果，并绘制图表。因此它不只是一个一次性脚本，而是一个可用于比较不同规划策略的小型实验框架。

## 2. 问题定义

输入包括一组候选景点、一个包含酒店节点的交通时间矩阵，以及旅行配置。每个景点包含 ID、名称、类别、二维坐标、费用、参观时长、开放时间、关闭时间和优先级分数。旅行配置包括总预算、旅行天数、每日开始和结束时间，以及可接受的最大等待时间。

目标是生成一个多天行程，使所有访问都满足硬约束，并尽量最大化总满意度：

``` text
total_satisfaction = sum(priority_score for all visited attractions)
```

这个问题可以看作带时间窗和预算限制的多天 Orienteering Problem。真正的难点在于多种资源约束会相互耦合。某个景点单独看似乎值得去，但一旦考虑到从当前位置前往的交通时间、提前到达后的等待时间、剩余预算，以及当天结束前必须返回酒店等要求，它就可能不再可行。由于精确优化的代价较高，而且会明显增加实现复杂度，本项目采用启发式搜索方法来构建一个可运行、可解释、可评估的原型。

## 3. 约束与假设

规划器检查以下硬约束：

| 约束              | 说明                             |
|-------------------|----------------------------------|
| budget            | 景点费用不能超过剩余预算         |
| opening hours     | 访问必须在景点关闭前完成         |
| max waiting       | 提前到达后的等待时间不能超过限制 |
| day window        | 每天结束前必须能返回酒店         |
| no duplicates     | 同一景点不能重复访问             |
| known attractions | 行程中的景点必须存在于数据集中   |

可行性检查器会对每个候选景点逐一应用这些约束。它先排除超出剩余预算的景点，再根据当前位置计算到达时间；如果景点尚未开放，则加入等待时间；之后检查访问是否会在关闭前完成；最后验证游客是否仍能在当天结束时间前回到酒店。也就是说，可行性是在构造过程中直接保证的，而不是先生成再事后修补。

主要建模假设包括：所有数据均为合成数据；交通时间由二维距离和固定速度计算；每天都从同一家酒店出发并返回；开放时间每天相同；用户偏好分数固定；系统只进行离线规划，不处理实时拥堵、延误或临时关闭。

## 4. 合成数据设计

`src/data_generator.py` 生成六类景点：museum、park、restaurant、landmark、market 和 entertainment。不同类别具有不同的费用范围、参观时长、开放时间、关闭时间和分数偏置。例如，entertainment 通常更贵、停留时间更长；landmark 往往参观时间较短但价值较高；market 虽然便宜，但关闭时间通常更早。这样一来，规划器面对的是带有真实权衡关系的候选集，而不是一组几乎相同的点。

景点位置不是完全随机均匀分布的，而是围绕若干聚类中心生成，以模拟真实城市中景点常常按片区聚集的现象。这一点很重要，因为路线规划质量与空间结构密切相关；如果景点完全均匀随机，交通模式会变得不够真实，也更难观察到“更会搜索的算法”带来的优势。酒店固定在地图中心附近，交通矩阵包含 `"hotel"` 和所有景点 ID。

交通时间通过欧氏距离除以固定速度 30 km/h 计算，再转换成分钟。虽然这是简化模型，但它会生成一个对称、易解释、适合受控实验的交通时间矩阵。数据生成器带有随机种子，因此实验具有可复现性。

运行单场景时，系统会把数据保存到：

``` text
data/<scenario>/attractions.csv
data/<scenario>/travel_matrix.csv
```

## 5. 规划算法

### 贪心基线

`plan_itinerary` 在每一步都从当前所有可行候选中选择综合分数最高的景点。综合分数由三个归一化后的部分组成：

``` text
composite_score =
    0.40 * normalized(priority_score)
  + 0.35 * normalized(time_efficiency)
  + 0.25 * normalized(cost_efficiency)
```

其中：

``` text
time_efficiency = priority_score / ((travel + wait + duration) / 60)
cost_efficiency = priority_score / max(cost, 1.0)
```

这使得贪心基线比“只挑最高分景点”更有意义。它仍然优先考虑景点本身的吸引力，但也会惩罚那些为了获得同样价值却要消耗过多时间或预算的候选。该方法速度快、易于解释，但只看当前一步，因此可能过早消耗预算或时间。

### Beam Search

`plan_itinerary_beam_search` 会同时保留前 `K` 个部分行程状态。每个状态保存：已经完成的天数、当前这一天的部分行程、当前时间和位置、已访问集合以及剩余预算。每一轮它都会扩展所有可行的下一步访问，然后只保留得分最高的 `K` 个状态。

状态评分综合了多个目标：

``` text
state_score =
    1.0 * total_satisfaction
  + 2.0 * num_attractions
  + 3.0 * travel_efficiency
  + 8.0 * category_diversity
  + 0.15 * remaining_budget
```

因此，beam search 比贪心算法更不容易“只顾眼前”。它可以同时保留多条有希望的部分路径，而不是立刻把所有选择压在一条路线上。不过它依然是启发式方法而不是最优求解器，所以行为会受到权重设计的影响。`tight_time` 场景就很好地说明了这一点：当真正稀缺的是时间而不是预算时，奖励“保留预算”的启发式会带来负面影响，使搜索偏向更便宜但不一定更高效的访问组合。

Beam search 和贪心算法共享同一套硬约束检查函数。两者的差异只在搜索策略，而不在约束定义，因此比较更公平。

## 6. 原型实现

主要代码结构如下：

``` text
src/
  data_generator.py     synthetic attractions and travel matrix
  scenarios.py          five predefined experiment scenarios
  planner.py            constraints, greedy planner, beam search planner
  evaluator.py          feasibility validation and metric calculation
  formatter.py          optional readable itinerary printing
  visualizer.py         matplotlib charts
  experiment_runner.py  scenario and algorithm sweep
  main.py               argparse command-line entry point
```

实现整体是一个流水线。`main.py` 负责解析命令行参数，并分发到单次运行或完整实验。`data_generator.py` 负责创建合成环境，`planner.py` 负责构造行程，`evaluator.py` 负责独立验证结果并计算指标，`experiment_runner.py` 负责在多个场景和算法上重复这一流程，并写出 `outputs/experiment_results.csv`、Markdown 行程文件和图表。

一个很有价值的设计点是“验证”和“规划”是分离的。规划器会尽量通过构造保证正确，但 `validate_itinerary` 仍会独立复查最终行程，检查是否出现超预算、等待过长、访问结束晚于关门时间、无法按时回酒店、重复访问、天数超限、未知景点 ID 等问题。这能降低“规划器自身的 bug 悄悄污染实验结果”的风险。

常用命令：

``` bash
python src/main.py
python src/main.py --show-itinerary
python src/main.py --run-experiments
python src/main.py --make-charts
python -m unittest discover -s tests -v
```

## 7. 实验场景

| 场景          | 景点数 | 天数 |  预算 | 每日时间窗  | Beam 宽度 | 目的       |
|---------------|-------:|-----:|------:|-------------|----------:|------------|
| easy          |     20 |    3 | \$400 | 08:00-22:00 |         5 | 宽松约束   |
| standard      |     30 |    3 | \$200 | 09:00-21:00 |         5 | 标准参考   |
| tight_budget  |     30 |    3 |  \$80 | 09:00-21:00 |         5 | 预算紧张   |
| tight_time    |     30 |    3 | \$200 | 10:00-17:00 |         5 | 时间紧张   |
| large_problem |     50 |    5 | \$400 | 09:00-21:00 |        10 | 扩展性测试 |

所有默认实验都使用随机种子 42。这些场景共同构成了一个有代表性的测试范围：既有约束很宽松、两种算法都应表现不错的情况，也有单一资源明显成为瓶颈的情况，还有一个更大的问题规模用于考察运行时间和搜索宽度带来的影响。

## 8. 评估指标

系统首先使用 `validate_itinerary` 独立检查可行性，然后再计算指标。主要指标包括：是否可行、违反约束数量、访问景点数、总花费、剩余预算、总满意度、总参观时间、总交通时间、交通占比、类别多样性、每日均衡度和运行时间。

其中有两个派生指标尤其值得说明：

``` text
travel_ratio = total_travel_minutes / (total_travel_minutes + total_visit_minutes)
daily_balance = max(1 - stdev(visits_per_day) / mean(visits_per_day), 0)
```

`travel_ratio` 衡量旅行者的有效活动时间中有多少比例花在移动上而不是参观上，通常越低越好。`daily_balance` 衡量景点访问是否较均匀地分布在不同天数上，越接近 1.0 表示分布越均衡，越低则表示行程前置或某些天利用不足。

之所以需要这一组指标，是因为单一数字很难完整描述行程质量。某个算法可能通过大量交通换来更高满意度，也可能把预算用得很“满”，却生成一个极不均衡的计划。因此评估必须同时看质量、效率、多样性和实际可执行性。

## 9. 实验结果

结果来自 `outputs/experiment_results.csv`：

|----|----|---:|---:|---:|---:|---:|---:|---:|---:|---:|----|---:|
| easy | greedy | 20 | \$352 | \$48 | 116.3 | 1680 | 308.0 | 0.155 | 1.00 | 0.62 | OK | 0.0158 |
| easy | beam_search | 20 | \$352 | \$48 | 116.3 | 1680 | 302.5 | 0.153 | 1.00 | 0.62 | OK | 0.1632 |
| standard | greedy | 18 | \$200 | \$0 | 106.4 | 1230 | 230.0 | 0.158 | 1.00 | 0.40 | OK | 0.0233 |
| standard | beam_search | 19 | \$198 | \$2 | 110.4 | 1305 | 277.3 | 0.175 | 1.00 | 0.67 | OK | 0.2279 |
| tight_budget | greedy | 10 | \$80 | \$0 | 64.4 | 600 | 135.4 | 0.184 | 0.67 | 0.00 | OK | 0.0140 |
| tight_budget | beam_search | 11 | \$80 | \$0 | 66.2 | 735 | 168.5 | 0.186 | 0.83 | 0.04 | OK | 0.1184 |
| tight_time | greedy | 15 | \$199 | \$1 | 98.3 | 1005 | 225.6 | 0.183 | 0.83 | 0.80 | OK | 0.0232 |
| tight_time | beam_search | 14 | \$163 | \$37 | 94.6 | 1020 | 213.5 | 0.173 | 1.00 | 0.88 | OK | 0.1870 |
| large_problem | greedy | 35 | \$400 | \$0 | 226.2 | 2340 | 441.2 | 0.159 | 1.00 | 0.41 | OK | 0.0714 |
| large_problem | beam_search | 37 | \$387 | \$13 | 226.5 | 2355 | 554.9 | 0.191 | 1.00 | 0.65 | OK | 2.2351 |

主要结论如下：

1.  Beam search 在 `standard`、`tight_budget` 和 `large_problem` 中优于贪心算法，在 `easy` 中与贪心打平。
2.  `tight_time` 是一个明显例外。由于真正稀缺的是时间而不是预算，beam search 中偏好保留预算的启发式产生了副作用，导致访问数和满意度反而下降。
3.  Beam search 通常能生成更均衡的每日安排，例如 `standard` 场景中的每日均衡度从 0.40 提升到 0.67。
4.  Beam search 的运行时间更高，但在 20 到 30 个景点规模下仍低于 1 秒；在 `large_problem` 中约为 2.24 秒。
5.  所有实验组合都满足硬约束，没有可行性违规。

如果进一步分场景理解，结论会更清楚。在 `easy` 中，两种算法都访问了全部 20 个景点，满意度也完全相同，说明在约束足够宽松时，更复杂的搜索策略带来的额外收益很有限。在 `standard` 中，beam search 多访问了 1 个景点，并提升了约 4 分满意度，说明在中等资源压力下，保留多条部分路径确实有帮助。在 `tight_budget` 中，beam search 的提升幅度不算大，但仍然多安排了 1 个景点，并改善了类别多样性，说明它更擅长在有限预算下跨天分配低成本高价值候选。在 `large_problem` 中，虽然满意度提升不大，但 beam search 仍多访问了 2 个景点，并显著改善了每日均衡度，这说明随着搜索空间增大，更宽的搜索会开始体现价值。

`tight_time` 的结果尤其重要，因为它暴露的是局限而不是成功。Beam search 在这个场景下的交通占比更低、均衡度更高，但访问数和满意度却更差。这说明它并不是“整体无效”，而是当前启发式与该场景的主导瓶颈不匹配。这个失败案例很有价值，因为它直接指向下一步改进方向：当时间窗比预算更紧时，状态评分应该自适应地降低“保留预算”的权重。

## 10. 可靠性与局限性

项目在 `tests/test_smoke.py` 中提供了 smoke tests。这些测试覆盖了：数据生成可复现性、交通矩阵对称性、主要可行性失败模式、规划器输出结构、评估器关键字段，以及两种算法在标准场景下的端到端可行性。对于一个课程原型而言，这已经能够提供基本可信度，但还不能等同于穷尽式验证。

本项目的主要局限包括：只使用合成数据；交通模型较为简化；用户偏好分数固定而非学习得到；没有实时重规划；beam search 使用固定启发式权重；没有与精确最优解进行比较；测试主要是代表性的 smoke coverage，而不是覆盖所有边界情况。另外，当前默认实验基本只使用每个场景的一个随机种子，因此报告中的结果更像“稳定的示例结果”，而不是对多次随机实例取均值后的统计结论。

## 11. 未来改进

未来可以加入自适应启发式权重、真实地图数据、随机交通时间、用户偏好学习、小规模 ILP 最优基线、多目标搜索、日内局部优化，以及简单的 Web 界面。另一个很值得做的扩展是对多个随机种子重复实验，这样报告中就能给出平均表现和方差，而不只是每个场景的一次确定性运行结果。

## 12. 结论

本项目完成了一个端到端的受约束旅行行程规划原型，包括合成数据生成、硬约束建模、贪心基线、beam search 规划器、独立验证器、实验运行器和图表可视化。实验结果表明，beam search 通常能够提升行程质量和每日均衡度，但在“时间而非预算”成为主导约束时，也可能被固定启发式误导。因此，这个原型既是一个可工作的规划系统，也是一个可用于比较不同搜索策略的小型实验平台。

总体来看，项目达到了 final project 对“构建 AI-powered prototype、分析规划问题、评估多场景表现、清晰说明自身代码贡献”的要求。
